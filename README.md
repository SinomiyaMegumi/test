# 《项目启动书》（Project Charter）

## 1. 项目名称
**实验室预约管理系统（Lab Reservation System）**

## 2. 项目背景与意义
### 2.1 背景
学校实验室资源（机房、创新实验室、电子/化学实验室等）通常存在以下痛点：  
- 预约流程依赖线下登记或分散沟通，信息不透明  
- 容易出现同一时间段重复预约、冲突占用  
- 管理员统计与审核成本高，记录查询效率低  
- 预约痕迹难追溯，责任界定不清晰  

### 2.2 意义
本项目旨在构建一套**在线、规范、可追踪**的实验室预约系统，实现：  
- **高效预约**：师生自助选择实验室/日期/时间段提交预约  
- **冲突避免**：同一实验室同一时间段不可重复预约，保证一致性  
- **透明管理**：预约记录清晰可查，支持审核/拒绝与统计筛选  
- **无纸化流程**：减少线下沟通与登记，提升管理效率与用户体验  

## 3. 项目目标与范围
### 3.1 目标（交付物导向）
- **用户端**：注册登录、实验室列表、在线预约、我的预约（可取消）、个人中心  
- **管理员端**：实验室管理、预约审核（可配置为自动审批）、预约记录查询、用户管理（禁用）  
- 满足基本非功能需求：安全性、并发一致性、响应时间、易用性

### 3.2 范围边界（不做/后续可扩展）
- 暂不接入统一身份认证（如校园 CAS/SSO），先使用学号/工号注册  
- 暂不做短信/邮件通知（可作为后续迭代）  
- 暂不做复杂权限体系（先区分用户/管理员两类角色）  
- 暂不做实验室设备借用/耗材管理（后续可扩展模块）

## 4. 关键干系人（Stakeholders）
- **普通用户**：学生、教师（预约与管理个人预约）  
- **管理员**：实验室管理员/教务相关人员（维护实验室信息、审核与统计）  
- **项目团队**：产品/前端/后端/测试与运维协作交付

## 5. 技术选型与总体架构
- **后端**：Java + Spring Boot（RESTful API）  
- **前端**：Vue3（Web 管理端与用户端界面，可同一项目多页面/多角色）  
- **数据库**：MySQL  
- **版本控制**：Git（建议 GitFlow 或 trunk-based 简化流程）  
- **安全措施（基本）**：密码加密存储（BCrypt）、参数校验、防 SQL 注入（ORM/预编译）、鉴权（JWT/Session 二选一）

## 7. 核心功能模块列表（Backlog）
> 以“用户故事/模块”粒度列出，可用于后续拆分迭代与任务分配。

### 7.1 用户端 Backlog
- **账号体系**
  - 学号/工号注册、登录、退出
  - 密码修改、个人信息查看
- **实验室信息**
  - 实验室列表展示（名称、地点、容量、设备描述等）
  - 实验室详情页（可选）
- **预约功能**
  - 选择实验室、日期、时间段（上午/下午/晚上）提交预约
  - 预约冲突提示（同实验室同时间段不可重复）
  - 预约状态展示（待审核/已批准/已拒绝/已取消）
- **我的预约**
  - 查询个人预约记录（分页/筛选可选）
  - 取消未开始的预约（带确认弹窗）

### 7.2 管理员端 Backlog
- **实验室管理**
  - 添加/编辑/删除实验室信息
  - 实验室列表管理、条件筛选（可选）
- **预约审核**
  - 查看预约申请列表
  - 批准/拒绝（拒绝理由可选）
  - 审核开关（可配置：自动批准/人工审核）
- **预约记录查询**
  - 按实验室、日期、时间段、状态等条件查询
  - 导出（可选：CSV/Excel，后续迭代）
- **用户管理**
  - 用户列表查看
  - 禁用/启用账号

### 7.3 非功能性 Backlog（必须项）
- **安全**
  - 密码加密存储（BCrypt）
  - 基础防注入与参数校验
  - 登录态与接口鉴权
- **并发一致性**
  - 预约唯一性约束（实验室+日期+时间段）
  - 事务一致性与错误提示
- **性能与体验**
  - 常用列表分页
  - 接口响应 < 3 秒（合理索引、避免 N+1 等）
